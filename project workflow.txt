Le workflow proposé : ( Plateforme de gestion des licences avec intégration de l’IA) Voici le flux complet que tu dois implémenter (sous forme de pipeline d’états) : 1. Client demande une licence → statut = requested 2. Notification envoyée à l’admin (via email, websocket, ou système de notif) 3. Admin passe le statut à in_progress 4. Paiement : hors plateforme (négocié directement entre admin et client) → pas ta responsabilité 5. Admin confirme et passe le statut à completed (ou active / valid) 6. Le système génère automatiquement une licence valide 7. Génération de clés asymétriques (private + public key) 8. Retourner la public key à l’admin 9. L’admin insère la public key dans l’application "master data" (l’application dédiée du client, celle qui contient les données maîtres) 10. Client reçoit une notification : « Ta licence est générée, tu peux maintenant ouvrir l’application et travailler » 11. Résultat final pour le client → il reçoit/utilise la public key (ou un token signé avec elle) Ce pipeline doit être implémenté dans le backend avec des API professionnelles, des statuts bien gérés en base de données, et des transitions d’état contrôlées (tu ne peux pas passer directement de requested à completed sans passer par in_progress, par exemple). La partie sécurité avec clés asymétriques (private / public key) + JWT Tu as bien compris le schéma global, mais on va le préciser : Le License Service génère une paire de clés asymétriques (typiquement RSA ou ECDSA) pour chaque licence ou par tenant/application. Private key : reste secrète, utilisée uniquement par le License Service pour signer les tokens/licences. Public key : est distribuée et insérée dans l’application cliente ("detected_app" ou "master data"). Quand l’application cliente veut vérifier si une feature est autorisée ou valider la licence : Elle envoie à l’application master data : app_code + client_code La master data contacte le License Service (ou utilise le token signé). Le License Service signe un JWT (JSON Web Token) avec la private key. L’application cliente vérifie la signature du JWT avec la public key (sans avoir besoin d’appeler le serveur à chaque fois → mode offline-friendly). Pourquoi asymétrique + JWT ? Le JWT contient les infos de la licence (plan, features, limites, expiration…). La signature avec private key garantit que personne ne peut falsifier la licence. La public key permet à l’application cliente de vérifier l’authenticité localement (très performant, fonctionne même offline). C’est exactement le modèle recommandé dans le cahier des charges (Option B : Signed License Token).
